# Task P3.A — Logistic loss, IRLS, logistic assumptions

## What's built

* `LogisticLoss` class and `logistic` singleton in `model_crafter/loss.py`,
  exposing `value` / `gradient` / `hessian` / `link`. The aggregated loss
  is `(1/sum w) * sum w_i * (log(1+exp(eta_i)) - y_i*eta_i)` (matches the
  `_SquaredErrorLoss` normalisation convention). `link(eta) = expit(eta)`
  is numerically stable for `|eta|` up to ~1e3. Assumptions tuple
  declares `FullRankDesign`, `BinaryOrProportionTarget`,
  `NoPerfectSeparation` (HARD); `ClassBalance`, `CoefficientStability`,
  `PredictiveStability` (SOFT); `LinkAdequacy` (INFO opt-in).
* `model_crafter/assumptions/logistic.py` — four checks:
  * `BinaryOrProportionTarget` (HARD) — target in `{0,1}` or `[0,1]`.
  * `NoPerfectSeparation` (HARD, post-fit) — fires when either
    `max(|slope|) > 1e3` *or* the saturated-probability fraction
    (`p < 1e-4` or `p > 1 - 1e-4`) exceeds 95 %. The probability-
    saturation signal compensates for the fact that the post-fit
    `fitted_stub` in `solve/__init__.py` doesn't carry `converged`.
    Failure message embeds the verbatim
    `"penalty=mc.l2(...) is the standard remedy for perfect separation (ESL §4.4.2)."`.
  * `ClassBalance` (SOFT, default `min_minority=0.01`).
  * `LinkAdequacy` (INFO) — decile Hosmer-Lemeshow chi-squared using only
    numpy + scipy; statsmodels is not a runtime dep.
* `model_crafter/solve/irls.py`:
  * `solve_logistic_irls` for `(LogisticLoss, NoPenalty)` — standard IRLS
    via QR factorisation of `sqrt(W) X`; returns closed-form SE from
    `sqrt(diag((X' W X)^{-1}))`.
  * `solve_logistic_ridge_irls` for `(LogisticLoss, L2Penalty)` — same
    loop with an augmented Tikhonov design that zeros the intercept row.
    `coefficient_se = None` (bootstrap is the recommended tool).
  * `solve_logistic_prox_cd` for `(LogisticLoss, L1Penalty)` and
    `(LogisticLoss, PenaltySum)` — FHT 2010 §2.6 proximal-Newton / CD:
    the outer IRLS loop builds `(W, z)`, the inner step dispatches into
    `coordinate_descent_path`. Penalty parameters are rescaled by
    `n / sum(working_w)` to compensate for the CD solver's internal
    weight normalisation to sum-to-n (without this rescaling, the
    objective the CD solver minimises differs from the outer Taylor
    expansion and the fit converges to a different optimum).
* Tiny additive exports of `BinaryOrProportionTarget`, `ClassBalance`,
  `LinkAdequacy`, `NoPerfectSeparation` in
  `model_crafter/assumptions/__init__.py` (just the two import block and
  `__all__` entries — no behaviour changes).
* Tests in `tests/test_logistic.py` (118 tests; 117 pass + 1 xfail) and
  `tests/test_irls.py` (12 pass). Total P3.A test count: 130 pass + 1
  xfail. The xfailed test documents what P3.INTEG must wire (see below).

## What's tested

* **Acceptance #1 — coefficients to 1e-6.** `LogisticLoss + NoPenalty`
  reproduces `statsmodels.GLM(family=Binomial()).fit().params` to atol
  1e-6 on the **Spector & Mazzeo 1980** dataset
  (`statsmodels.datasets.spector` — 32 rows, binary `GRADE` target,
  predictors `GPA`, `TUCE`, `PSI`) and on a 2000-row synthetic dataset
  with known coefficients. SEs match to atol 1e-4.
* **Acceptance #2 — KS / AUC to 1e-9.** Inline AUC (Mann-Whitney form +
  trapezoid-of-ROC cross-check) matches both ways to 1e-9, and KS
  matches `scipy.stats.ks_2samp` to 1e-9.
* **Acceptance #3 — separation raises.** `y = (x > 0)` raises
  `AssumptionError("NoPerfectSeparation: ... penalty=mc.l2(...) is the
  standard remedy for perfect separation (ESL §4.4.2).")`. Adding
  `penalty=l2(0.5)` to the same data fits successfully.
* **Acceptance #4 — predict ∈ [0, 1] property test.** 100 random seeds;
  probabilities computed via `1/(1+exp(-X@beta))` (inline) all lie in
  `[0, 1]`. A companion `xfail` test calls `mc.predict(sol, data)`
  directly with a clear xfail reason for P3.INTEG.
* IRLS: weighted variant matches `sm.GLM(var_weights=w)`; ridge-IRLS
  matches `sm.GLM.fit_regularized(L1_wt=0, alpha=lam vector)` to atol
  5e-3; lasso and elastic-net IRLS match `fit_regularized(L1_wt=1)` and
  `(L1_wt=in_between)` to atol 2e-2. The lasso path drives noise
  coefficients to exact zero (sparsity confirmed).
* Convergence flag, iteration count, and `tol` are exposed via
  `solver_info`.

## Open question for P3.INTEG (REQUIRED WIRING)

`mc.predict(sol, new_data)` in `model_crafter/solve/__init__.py`
currently returns `X @ beta` (the linear predictor `eta`). For logistic
specs the package guarantees a probability output (DESIGN.md §3.3).
**At P3.INTEG, please update `predict()` (one-line change) to apply the
loss's `link()` method when present:**

```python
beta = sol.coefficients.reindex(list(design.columns)).to_numpy(dtype=float)
eta = design.values @ beta
link = getattr(sol.spec.loss, "link", None)
yhat = link(eta) if callable(link) else eta
return pd.Series(yhat, index=new_data.index, name=sol.spec.target)
```

When this lands, the existing `tests/test_logistic.py::test_predict_returns_probabilities_for_logistic`
will flip from xfail to pass (it asserts `(yhat >= 0).all() and (yhat <= 1).all()`).

## Other notes

* The `irls` module self-registers four entries in the solver registry
  on import. The integration agent must trigger that import — the
  current pattern in `solve/__init__.py` is the alphabetical list of
  ``from model_crafter.solve import {coordinate, ols, ridge}``; add
  ``irls`` to that list. The tests in `tests/test_logistic.py` and
  `tests/test_irls.py` import `model_crafter.solve.irls` explicitly so
  they're independent of that wiring.
* The proximal-Newton CD uses an implicit warm start (it always starts
  CD from zero on each outer iteration, relying on the working response
  to embed the current iterate). A real warm-start hook on
  `coordinate_descent_path` would speed up convergence for large
  problems — slated for Phase 6 / profiling.
* Standard errors for the ridge-logistic case are intentionally
  `None`. `mc.bootstrap` (Task P3.C) is the recommended uncertainty
  tool, per DESIGN.md §3.2.3.
* `LinkAdequacy` (INFO) uses a Hosmer-Lemeshow decile chi-squared
  written in numpy + scipy; statsmodels is *not* imported at runtime
  (only in tests, per the rule).
