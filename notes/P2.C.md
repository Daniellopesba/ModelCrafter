# Task P2.C — Coordinate descent for lasso and elastic net

## What got built

`model_crafter/solve/coordinate.py` implements a cyclic coordinate-descent
solver for the elastic-net least-squares objective
`(1/2n) ||y - Xβ||² + λ₁ Σ|β_j| + (λ₂/2) Σβ_j²` (ESL §3.4.2 / 3.4.3, FHT
2010 eq. 5). The kernel `coordinate_descent_path(X, y, lambdas_l1, lam_l2,
weights, intercept, tol, max_iter, warm_start)` is the warm-started entry
point: it standardises `X` internally with weighted moments (the `glmnet`
convention), centres `y`, runs sweep-based soft-thresholding updates, then
de-standardises coefficients back to the original `X` scale and recovers
the intercept from `ȳ - μ_X · β`. Sample weights are normalised internally
to sum to `n` so the same `λ` has the same meaning for any positive weight
scaling; this matches statsmodels' WLS+elastic-net rescaling. **Naive**
updates rather than covariance updates — documented in the module
docstring with a pointer to FHT 2010 §2.2; the crossover to covariance
updates is straightforward when profiling shows it matters. The dispatch
entry points `solve_lasso_cd` (for `(squared_error, L1Penalty)`) and
`solve_enet_cd` (for `(squared_error, PenaltySum[L1+L2])`) both route to
the kernel. Registration is wrapped in a `register_solvers()` function
that imports `L1Penalty` / `PenaltySum` from `model_crafter.penalty` and
is a no-op if they're not yet present — so this module loads on the
pre-P2.A branch and the integration agent re-calls `register_solvers()`
once the penalty types are merged. `model_crafter/validation/__init__.py`
exposes the lambda-path helpers; `lambda_path.py` ships `log_grid(low,
high, n)` (descending log-spaced grid), `compute_lambda_max(X, y, *,
alpha)` (the KKT formula `max_j |x_jᵀy| / (nα)` on standardised X /
centred y), and `lambda_path(spec, data, n, ratio)` which extracts α from
the spec's penalty and returns `[λ_max, …, ratio·λ_max]`.

## What's tested

`tests/test_lambda_path.py` (9 tests) and `tests/test_coordinate.py`
(15 tests) — all green. The five acceptance criteria from AGENTS.md
Task P2.C:

1. **Match a glmnet-style reference to 1e-6 across a lambda path** for
   both lasso (`test_prostate_lasso_path_matches_glmnet_style_reference_to_1e_6`)
   and elastic net at α=0.5 (`test_prostate_elastic_net_path_…`). The
   reference is statsmodels `OLS.fit_regularized(method="elastic_net",
   cnvrg_tol=1e-14, zero_tol=0.0)` fit on internally-weighted-standardised
   X (the same internal representation our CD uses) with the de-
   standardisation reapplied to recover original-scale coefficients.
   statsmodels' docstring states the implementation "closely follows the
   glmnet package in R"; the parameterisation match was cross-validated
   by checking that at λ=0 statsmodels recovers OLS to 1e-14 and at
   λ≥λ_max returns β=0.
2. **Warm-started path is measurably faster.** The benchmark in
   `test_warm_started_path_is_faster_than_from_scratch` runs a 50-point
   λ path on an n=600, p=100 problem 5 times in each mode and takes the
   min. Locally observed ~1.5–1.9× speedup with `tol=1e-8`; the test
   asserts `> 1.25×` as a CI-friendly guard with the actual ratio printed
   on failure.
3. **λ ≫ λ_max → β=0; λ=0 → OLS to 1e-8.** Both verified directly;
   `test_lasso_at_lambda_zero_recovers_ols_to_1e_8` matches statsmodels'
   OLS to ~1e-12.
4. **α=1 reproduces the lasso solver; α=0 matches the ridge closed form
   to 1e-6.** Both verified, ridge match to ~1e-16 in fact.
5. **Weighted CD matches a hand-derived weighted-orthonormal reference
   to 1e-8.** `test_weighted_cd_matches_hand_derived_orthonormal_solution`
   constructs a design with `XᵀWX/n = I` and asserts the CD output equals
   `S(XᵀWy/n, λ)` — closed-form check, no third-party optimiser.

Dispatch wiring is exercised through `_make_inputs` (a SolverInputs
factory) using **local penalty stubs** (`L1Penalty`, `L2Penalty`,
`PenaltySum` dataclasses with the documented contract — `.lam` and
`.parts`). The stubs sit in the test file so the tests are independent
of P2.A's merge state. Edge cases covered: `PenaltySum` with two L1
parts raises `TypeError`, pure-L2 `PenaltySum` is rejected with a
pointer to the ridge solver, L1-only `L2Penalty` routed to the lasso
entry point raises with a clear message.

`ruff check .` (clean), `pyright --pythonpath …/.venv/bin/python`
(0 errors), `pytest -q` (120 passed) all green.

## Stubs / mocks / open questions

* **P2.A interface dependency.** `coordinate.py` registers via duck-typed
  attribute access on whatever objects `model_crafter.penalty` exports —
  no imports of `L1Penalty` / `L2Penalty` are required at module import
  time. The `register_solvers()` function tries to import them and is a
  no-op if absent. Integration step: P2.INTEG should add a call to
  `model_crafter.solve.coordinate.register_solvers()` from
  `model_crafter/solve/__init__.py` after the P2.A merge. The tests use
  local stubs whose class names match P2.A's (`L1Penalty`, `L2Penalty`,
  `PenaltySum`) and whose contracts match the AGENTS.md spec; once P2.A
  merges, the stubs in the test file can be deleted and the real types
  imported instead.

* **Standardisation convention.** Coefficients are reported on the
  **original** `X` scale (not the internally-standardised scale).
  Documented prominently in the module docstring; users comparing to
  `glmnet` output should expect a match because glmnet does the same.
  The alternative — exposing standardised coefficients — was rejected
  because credit-risk users want coefficients in the units they ship to
  production.

* **Naive vs covariance updates.** Picked naive. Trade-off documented in
  the module docstring (FHT 2010 §2.2). Crossover to covariance is a
  Phase-6 optimisation if profiling demands.

* **Lambda-path helpers location.** `model_crafter/validation/__init__.py`
  is new and currently only re-exports `lambda_path` and `log_grid`. The
  rest of the validation subpackage (splitters, `cross_validate`, `tune`,
  `bootstrap`) lands in Phase 3 (P3.B / P3.C). The `__init__.py` here is
  the minimal scaffold the P2.C deliverable requires; Phase 3 agents will
  extend it.

* **No SE for penalised fits.** `SolverOutputs.coefficient_se = None`
  because closed-form SEs don't exist for lasso/elastic-net (ESL §7.11).
  The bootstrap (P3.C) is the right tool; that's already documented in
  the solver_info dict via the absence of an SE rather than a synthetic
  zero.

* **Did-not-converge path.** When `max_iter` is exhausted the
  `SolverOutputs.converged` field is `False` and `solver_info` carries
  `n_iter == max_iter` and `max_coef_change` (the size of the last sweep
  change), so callers can introspect. Documented and tested.

* **Workspace.** Files written: `model_crafter/solve/coordinate.py`,
  `model_crafter/validation/__init__.py`,
  `model_crafter/validation/lambda_path.py`,
  `tests/test_coordinate.py`, `tests/test_lambda_path.py`. Nothing
  outside the P2.C workspace was modified. `model_crafter/solve/__init__.py`,
  `model_crafter/solve/_registry.py`, `model_crafter/penalty.py`,
  P2.B's `model_crafter/solve/ridge.py` are all unchanged.
