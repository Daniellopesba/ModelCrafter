# Task P2.B — Closed-form ridge solver

## What I built

`model_crafter/solve/ridge.py` implements the closed-form ridge solver for
the `(squared_error, L2Penalty)` dispatch key. The math follows ESL §3.4.1
(eq. 3.44) and matches the `glmnet(alpha=0)` convention:

- Objective: `(1/2n) ||y - β₀ - Xβ||² + (λ/2) ||β||²`, intercept
  **unpenalised**.
- Weighted variant: `(1/(2Σw)) Σ wᵢ (yᵢ - β₀ - xᵢᵀβ)² + (λ/2)||β||²` with
  effective ridge scale `(Σw)·λ` instead of `n·λ` — same relationship the
  existing OLS solver in `_internal/linalg.py` uses between OLS and WLS.
- Implementation: profile out the intercept by centering `(X, y)` with the
  (weighted) means, stack `√((Σw)λ)·I` underneath the (weighted) centered
  design, solve via QR. Avoids forming `(XᵀX)⁻¹` and squaring the
  condition number. Intercept recovered from `β₀ = ȳ − x̄ᵀβ`.
- `λ=0` is handled without forming the empty augmented block — the
  augmented stack reduces to the OLS QR path and matches the OLS solver to
  1e-10 (verified in `test_zero_lambda_ridge_matches_ols`).
- Standard errors are intentionally **not** computed; `coefficient_se=None`
  with a docstring pointing at `mc.bootstrap` per DESIGN.md §3.2.3.

The solver self-registers for `(_SquaredErrorLoss, L2Penalty)` on import
when `L2Penalty` is available; if P2.A has not yet merged the registration
is skipped silently (the test harness re-registers against its local stub).

## What's tested (`tests/test_ridge.py`, 39 tests, all green)

The test docstring quotes the four acceptance criteria verbatim.

- **Criterion 1** (closed form on synthetic data, 1e-10): 25-case
  `parametrize` over 5 seeds × 5 lambdas; each compares against a numpy
  hand-derivation of `β = (XᵀX + nλI)⁻¹ Xᵀy` on centered data, plus a
  separate "shift `y` by a constant → intercept shifts by the same
  constant" invariant test that verifies the intercept is genuinely
  unpenalised.
- **Criterion 2** (prostate dataset across a lambda path, 1e-6): 7-case
  `parametrize` over `λ ∈ {1e-4, …, 100}` checking the solver against the
  hand-derived closed form on the standardized 67-row training subset.
  Plus an independent cross-check at `λ=0.5` against statsmodels solving
  the augmented Tikhonov design — this catches "right answer, wrong
  objective" bugs because the statsmodels path is a structurally
  different implementation of the same equation.
- **Criterion 3** (weighted ridge, 1e-10): closed-form weighted hand
  derivation; plus a parity test that array-weights and column-name
  weights yield identical solutions to 1e-14.
- **Criterion 4** (λ=0 → OLS, 1e-10): solves the same data with `NoPenalty`
  and `l2(0.0)` and asserts the coefficient vectors agree.
- Dispatch contract: `get_solver(squared_error, L2Penalty(lam=1.0))`
  returns the registered solver; `solver_info` carries the
  `ridge_closed_form_qr` label and the requested `lambda`.

## Reference source used for prostate acceptance

R / `glmnet` is unavailable in this environment, so the prostate
acceptance uses two complementary references:

1. **The closed-form math itself** computed in-test via numpy
   `np.linalg.solve` on the normal equations of the centered system.
   `glmnet(alpha=0, standardize=TRUE)` minimizes the same objective on
   standardized predictors with an unpenalized intercept; holding the
   solver to a 1e-6 tolerance against the equation it claims to minimize
   is the correctness check.
2. **An independent statsmodels Tikhonov fit** on the augmented
   `[X_centered; sqrt(nλ)·I]`, `[y_centered; 0]` design. This is a
   structurally different code path that solves the same ridge objective
   — useful for catching algebraic bugs that would survive a
   matches-its-own-numpy-baseline check.

The published ESL Table 3.3 ridge values are at *degrees-of-freedom*
parameters (df=5, df=7), not at named lambdas, so they aren't directly
usable as test references without solving for the corresponding λ —
which would just be solving the same equation a third time. Documented
this choice in the test module docstring.

## How I handled the L2Penalty dependency

P2.A hadn't merged when I started. The contract from AGENTS.md is
`L2Penalty(lam: float, assumptions: tuple, value, prox, __add__)`. The
solver module attempts `from model_crafter.penalty import L2Penalty`; on
`ImportError` it silently skips registration (the integration agent will
import `solve.ridge` after P2.A lands, and the import will succeed). The
test file mirrors this pattern: it tries the real import first, falls
back to a local frozen-dataclass stub that matches the documented
contract, and explicitly registers the stub key when running against the
stub. Both paths (real `L2Penalty` and stub) exercise identical solver
code.

## Open questions / contract concerns

- **`Penalty + Penalty` flattening when P2.A lands.** My module only
  reads `penalty.lam`, so a `PenaltySum` won't dispatch here unless the
  registry key explicitly registers `(_SquaredErrorLoss, PenaltySum)` —
  but that case is owned by P2.C's coordinate descent. Nothing to do on
  my side, but worth noting at integration.
- **Standard errors.** I deliberately return `coefficient_se=None`. If a
  future task wants closed-form ridge SEs via the hat matrix
  `H = X(XᵀX + nλI)⁻¹ Xᵀ`, the QR factorisation in `_ridge_qr_solve`
  has enough state to compute them; I'd add that as a follow-up rather
  than gold-plate Phase 2.
- **`Solution.assumptions` for ridge.** The solver returns through the
  normal `solve()` dispatch which runs the assumption framework on the
  output. `L2Penalty.assumptions = (ComparableFeatureScales(...),)` lives
  in P2.A; once it lands, scale checks will fire automatically on
  unscaled features — exactly the P2.INTEG acceptance criterion. No
  changes needed in this workspace.
