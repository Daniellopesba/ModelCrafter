# P6 вҖ” Segmentation, diagnostics, repr polish

Single-agent task. The three pieces вҖ” segmentation, solution inspection, and
`_repr_html_` polish вҖ” landed as separate, additive layers on top of the
Phase 5 integration.

## What got built

### Piece 1 вҖ” Segmentation (DESIGN.md В§3.4)

* `SegmentedSpec(by, base)` and `mc.segmented(by=, base=)` in
  `model_crafter/spec.py`. Frozen + slotted dataclass mirroring
  `LinearSpec`'s eager-validation style. The constructor is keyword-only.
* `SegmentedSolution(spec, segments, n_obs)` in `model_crafter/solution.py`,
  also frozen + slotted, with mapping-style ergonomics (`__getitem__`,
  `__iter__`, `__len__`, `.keys/.values/.items`).
* `model_crafter/solve/segmented.py` вҖ” new module exposing
  `solve_segmented` and `predict_segmented`. The dispatch in
  `solve/__init__.py` is extended with `@overload` decorators so the
  static type of `solve(LinearSpec, ...)` remains `Solution` and
  `solve(SegmentedSpec, ...)` is `SegmentedSolution`. Same for `predict`.
* Per-segment `solve`s call the *inner* `solve(spec.base, slice_df, ...)`,
  passing every keyword argument through (weights / on_violation / suppress
  / classical_inference / stability_splitter). Each segment's
  `Solution.assumptions` is its own вҖ” verified by the test that the three
  reports have distinct object IDs.
* `solve_segmented` transparently calls `fit_binnings(base, slice_df)`
  per segment when the base contains unfitted WoE / binned terms. This
  preserves the "single declarative spec" north-star from В§3.4 (no
  pre-fit ceremony required when segmenting).
* Predict-time routing: rows whose segment value is not in
  `sol.segments` get `NaN` and a single warning naming the unseen
  segments and total affected rows.
* `mc.bootstrap(sub_sol, seg_train)` works on each per-segment solution
  unchanged вҖ” verified end-to-end in `test_end_to_end_woe_logistic_segmented`.
* `mc.performance(sub_sol, seg_slice)` works on each per-segment solution
  unchanged вҖ” same test.

### Piece 2 вҖ” Solution inspection (DESIGN.md В§5)

In `model_crafter/inspect.py` (extension-only, no refactors):

* `coefficients(sol)` вҶ’ `DataFrame[estimate, std_error, z, p_value]`. OLS
  uses t-distribution with `n-p` df; logistic uses Wald (standard-normal).
  For lasso / enet (no closed-form SE) the SE / z / p columns are NaN; the
  docstring points at `mc.bootstrap`.
* `Diagnostics` + `diagnostics(sol, data)` вҖ” `residuals`, `leverage`,
  `cooks_distance`, `studentized_residuals`, `sigma2`. Closed-form only.
* `hat_matrix(sol, data)` вҖ” `H = X(X^T X)^{-1} X^T` for OLS,
  `H_О» = X(X^T X + nО»I')^{-1} X^T` for ridge with `I'` zeroing the
  intercept. Emits an `O(nВІ)` warning for `n > 5000`.
* `Influence` + `influence(sol, data)` вҖ” DFBETAS (closed-form leave-one-out
  formula with leave-one-out ПғМӮВІ), Cook's distance, leverage. Cross-checked
  against an explicit row-loop in `test_influence_dfbetas_consistent_with_leave_one_out_refit`.
* Lasso / elastic-net / logistic call to `diagnostics` / `hat_matrix` /
  `influence` raises `NotImplementedError` whose message mentions
  `mc.bootstrap`. Verified by tests.
* All four functions accept `(sol, data)`. The Solution does not persist
  X (DESIGN.md В§6.2 keeps the value compact), so the training frame must
  be passed in. The error message makes that explicit.

#### Hand-derived OLS reference (acceptance fixture)

`tests/test_diagnostics.py::_hand_derived_ols_problem` builds the
fixture with `n=10, p=3`:

* `x1 = [1, 2, вҖҰ, 10]`, `x2 = [2, 1, 4, 3, 6, 5, 8, 7, 10, 9]`, intercept added.
* `y = 1 + 2В·x1 вҲ’ x2 + Оө` with `Оө ~ Normal(0, 0.5ВІ)` (seed 0).
* The reference computes ОІ, H, leverages, residuals, ПғМӮВІ, Cook's distance,
  and studentized residuals in raw numpy from the closed forms.
* Package output matches the reference to `atol=1e-12` across all four
  quantities, plus `np.trace(H) == p=3`, `H @ H == H`, `H == H.T`.

## Open items / honest red flags

* `model_crafter/__init__.py` is **not** edited per the task brief
  ("integration agent wires the new re-exports"). My tests import the
  new symbols directly from `model_crafter.spec`,
  `model_crafter.solution`, `model_crafter.inspect`, and
  `model_crafter.solve.segmented`. The integration agent should
  re-export `segmented`, `SegmentedSpec`, `SegmentedSolution`,
  `coefficients`, `diagnostics`, `hat_matrix`, `influence`,
  `Diagnostics`, `Influence` from the top-level package.
* `solve()` and `predict()` now take a union argument
  (`LinearSpec | SegmentedSpec` / `Solution | SegmentedSolution`).
  Static typing is preserved via `@overload` stubs so every existing
  call site continues to type-check at the narrower type.
* `diagnostics`, `hat_matrix`, and `influence` require the training
  frame as a second argument. This deviates from the В§5 signatures
  printed in DESIGN.md (`diagnostics(sol)` etc.) but a Solution does
  not persist X вҖ” the explicit `data` argument is the least surprising
  contract that's actually realisable in v0. A TypeError raised on
  `diagnostics(sol)` (no data) explains this to callers.
* The segmented predict's groupby uses `observed=True` to skip empty
  Categorical levels вҖ” same convention as
  `performance_by_segment`. Mixed Categorical / non-Categorical
  segmentation columns work, but very-large-cardinality columns will
  produce a `SegmentedSolution` with many keys; a future task could
  add a `min_segment_size` guard at solve time.
* Per-segment bootstrap is *not* automatic. The acceptance criterion
  asks that `mc.bootstrap(sub_sol, seg_train)` works on each
  sub-solution; the end-to-end test does exactly that. A future
  task could add a top-level `mc.bootstrap(seg_sol, data)` that
  bootstraps each segment independently, but the contract for that
  is open (per-segment `n_boot`? aligned random_state?) so I left
  it out.
* `_repr_html_` reuses pandas' `to_html` for tabular sub-views.
  pandas emits a leading `<table border="0">` plus a `dataframe`
  class вҖ” we add our own `classes=...` so styles do not collide. The
  HTML is intentionally self-contained but un-styled; downstream
  notebooks can attach their own CSS via the `mc-*` class hooks.
