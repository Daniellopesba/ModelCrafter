# P5.C — Model comparison

## What was built

`model_crafter/performance/compare.py`:

* **`Comparison`** — frozen, slotted dataclass with two fields:
  * `reports: dict[str, PerformanceReport]` — one entry per named
    solution, built by delegating to
    `model_crafter.performance.performance(sol, data, weights=...)`.
  * `delong_pvalues: pd.DataFrame` — square, indexed and labelled by
    solution name, with `NaN` on the diagonal. Off-diagonal entries are
    the unweighted DeLong two-sided p-values from
    `model_crafter.metrics.classification.delong_test`, populated
    symmetrically.

  `__repr__` matches DESIGN.md §3.3's "Model comparison" layout — a
  header line with `n` and `n_events`, an explicit `baseline = '<name>'`
  hint, then a metric × solution table for AUC, KS, Brier, Log-loss, and
  PSI vs reference. The first solution in the input mapping is treated
  as the baseline; the Δ column reports the second solution's value
  minus the baseline, and the p-value column carries the
  baseline-vs-challenger DeLong p-value on the AUC row only. With three
  or more solutions the repr points to `.delong_pvalues` for the full
  pairwise table.

* **`compare(solutions, data, *, weights=None)`** — pure orchestration:
  one `performance()` call per solution, one `delong_test()` call per
  unordered pair. No metric is recomputed inside this module.

## How the weights / DeLong contract is handled

`delong_test` raises `NotImplementedError` for non-uniform weights
(P3.D's documented limitation; weighted DeLong per Pepe 2004 §5.2 is
out of scope for v0). `compare` probes the weight vector up front
(`_weights_are_uniform`): if the weights would be non-uniform we **skip
the DeLong loop entirely** and leave `delong_pvalues` filled with `NaN`,
including the diagonal. The per-solution `PerformanceReport`s are still
built (they handle non-uniform weights fine; only the AUC CI inside
each report falls back to `None` per P3.D), so the metric table in the
`__repr__` is still informative. The repr footer prints:

    DeLong unavailable with non-uniform weights (see Pepe 2004 §5.2).

Uniform-but-non-`None` weights (e.g. an all-`1.0` column) are accepted
and DeLong is computed — a regression test pins this behaviour. There
is also a defensive `try/except NotImplementedError` inside the pair
loop so an edge-case probe disagreement degrades gracefully rather than
raising.

## What's tested

`tests/test_compare.py`, 11 tests:

1. **Type structure** — `compare(...)` returns a `Comparison`; `reports`
   maps to `PerformanceReport` instances; `delong_pvalues` is a square
   2×2 `pd.DataFrame` with the right index/columns and NaN diagonal.
2. **Symmetry** — `df.loc["a", "b"] == df.loc["b", "a"]`.
3. **Acceptance #2 / #3 (DESIGN.md §8 Phase 5)** — the p-value at
   `df.loc["a", "b"]` is *bitwise-equal* (floating-point equality) to
   the standalone `mc.delong_test(sol_a, sol_b, data).p_value`, because
   `compare` calls the same primitive. The AUC difference computed from
   the per-solution `PerformanceReport`s matches `delong_test(...).diff`
   to 1e-12.
4. **Three-way comparison** — 3×3 square table, NaN diagonal, fully
   symmetric off-diagonal, each entry matches the standalone primitive.
5. **Non-uniform weights fallback** — `compare(..., weights="w")` with
   non-uniform `w` returns a Comparison whose `delong_pvalues` is fully
   NaN and whose reports are still built. No `NotImplementedError`
   escapes.
6. **Uniform-weight column** — an all-`1.0` weights column does *not*
   trigger the fallback; the DeLong p-value matches
   `delong_test(..., weights="w").p_value` exactly.
7. **`__repr__` layout** — contains the `Comparison` header, the
   observation count, the metric labels AUC / KS / Brier / Log-loss
   from DESIGN.md §3.3, the solution names, a Δ column, and a
   p-value/DeLong mention.
8. **`__repr__` weights footer** — when weights make DeLong unavailable
   the repr explicitly says so.
9. **Frozen** — assigning to `cmp.reports` raises (slots + frozen).

## Acceptance criteria — quoted in test docstrings

> Acceptance #2 (DESIGN.md §8 Phase 5): `mc.delong_test` matches the
> `pROC::roc.test` R reference to 1e-6 — verified at P3.D. P5.C wraps
> that primitive and surfaces the same p-value in
> `Comparison.delong_pvalues` to floating-point equality.

> Acceptance #3 (DESIGN.md §8 Phase 5): `mc.compare` on two models
> trained with different lambdas on the same data shows AUC differences
> consistent with `report_a.discrimination.auc.value -
> report_b.discrimination.auc.value` on the same test set, including a
> DeLong p-value that matches `mc.delong_test` directly.

Both are pinned in `tests/test_compare.py` —
`test_compare_delong_pvalues_match_delong_test_primitive` and
`test_compare_auc_difference_matches_reports`.

## For the integration agent (P5.INTEG)

1. **Public API re-exports.** Per AGENTS.md, P5.C does not edit
   `model_crafter/__init__.py` or `model_crafter/performance/__init__.py`.
   The integration agent should add:

   ```python
   from model_crafter.performance.compare import Comparison, compare
   ```

   to `model_crafter/performance/__init__.py` (`__all__` extended with
   `"Comparison"` and `"compare"`), and re-export `compare` from the
   top-level `model_crafter/__init__.py` alongside `performance`.

2. **No plot helper.** DESIGN.md §3.3 hints at `cmp.calibration` (an
   overlaid calibration-curve plot) and a `.plot()` method on
   `Comparison`. Phase 5 acceptance doesn't require these, and the
   §8 Phase 6 plan says plots are lazy matplotlib helpers added there.
   Left for P6 unless flagged.

3. **`Δ` column choice.** The §3.3 example shows a single Δ column
   (`baseline` → `challenger`). With three or more named solutions
   there is no single "challenger", so the Δ column shows the second
   name's value minus the baseline, and the footer note points the
   user to `.delong_pvalues` for the full pairwise table. If P5.INTEG
   wants a column-per-non-baseline `Δ`, the layout in
   `Comparison.__repr__` is the only place to change.

4. **Weighted DeLong remains unimplemented.** Same status as P3.D —
   `compare` cleanly degrades when weights are non-uniform, and the
   repr documents the omission. The underlying primitive
   (`Pepe 2004 §5.2`) is the natural follow-up.
