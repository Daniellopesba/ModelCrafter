# Task P1.B — Assumption framework + OLS-specific assumptions

## What I built

The assumption framework for `model_crafter`, end-to-end. Concretely:

- `model_crafter/assumptions/_types.py` — `Severity` enum, `CheckResult`
  (frozen, slotted), `Assumption` `Protocol` (attributes declared via
  `@property` so frozen-dataclass implementations satisfy the protocol),
  `AssumptionError`. Extracted into its own module to avoid a circular
  import between the package root and the concrete-assumption modules.
- `model_crafter/assumptions/__init__.py` — `AssumptionReport` (frozen,
  slotted) with `by_severity()`, `passed()`, and a layered repr;
  orchestration via `run_assumptions(...)` and `check_assumptions(...)`;
  re-exports of every public name in `__all__`.
- `model_crafter/assumptions/_common.py` — internal helpers
  (`materialise_design`, `get_residuals`, `design_column_names`). Built
  the design matrix locally so prerequisite checks can run *before* P1.A's
  `_internal/design.py` is integrated; once P1.A ships, this module is the
  drop-in seam to delegate.
- `model_crafter/assumptions/prerequisites.py` — `FullRankDesign` (HARD).
  Uses `np.linalg.matrix_rank` for the pass/fail decision and a
  left-to-right span sweep to *name* offending columns.
- `model_crafter/assumptions/stability.py` — `CoefficientStability` and
  `PredictiveStability` (SOFT). Stubbed per the AGENTS.md contract: when
  `cv is None` they return a `passed=True` "skipped" `CheckResult` (no
  silent failure per DESIGN.md §9). Real CV-backed paths are implemented
  but exercised lightly until P3.B pins the `CVResult` shape.
- `model_crafter/assumptions/classical.py` — `ResidualNormality`
  (Shapiro-Wilk under n<5000, Anderson-Darling above), `Homoscedasticity`
  (hand-rolled Breusch-Pagan: auxiliary OLS of e^2 on X, LM = n·R²
  against χ²_k), `Independence` (Durbin-Watson), `LowVIF` (1 / (1 - R²_j)
  via leave-one-out OLS regressions; the suggestion field is the verbatim
  ESL §3.4.1 string pinned by AGENTS.md).

Orchestration semantics (DESIGN.md §4.2):

- `on_violation='raise'` (the default) raises `AssumptionError` on HARD
  failure. SOFT failures always warn (never raise). INFO results are silent.
- `on_violation='warn'` downgrades HARD to a warning.
- `on_violation='ignore'` records everything and emits no warnings.
- `classical_inference=False` (default) drops INFO checks entirely before
  running — they are absent from the report. `classical_inference=True`
  runs them at INFO severity.
- `suppress=` accepts a tuple of assumption classes and removes them
  from the collected set.
- Skipped checks (no solution / no CV for a check that requires it) return
  a `passed=True` `CheckResult` with an explanatory message; the
  orchestrator never silently drops.

immutability, `Severity` values, `AssumptionReport` grouping/passed/repr,

## Contract I depend on from P1.A

The framework reads only the documented attributes — it works against
any object that exposes them:

- `spec.target: str`
- `spec.features: Iterable` — strings or `Term`-like objects with a
  `.name` attribute (the framework treats each term name as one design
  column for now; basis expansion will need P1.A's `_internal/design.py`
  in Phase 4).
- `spec.loss` — must expose `assumptions: tuple[Assumption, ...]`.
- `spec.penalty` (optional) — may expose `assumptions`.
- `spec.intercept: bool` — defaults to `True` when missing.
- `solution.coefficients: pd.Series` indexed by design-column name.
- `solution.coefficient_se: pd.Series | None`.
- `solution.design_columns: tuple[str, ...]`.

The tests build minimal frozen-dataclass stubs (`StubLoss`, `StubTerm`,
`StubSpec`, `StubSolution`) and fit OLS via `np.linalg.lstsq` to produce
a `StubSolution` — no dependency on P1.A's concrete `LinearSpec` /
`Solution` types.

## Open questions / red flags

1. **`pyright` config gap.** `.venv/bin/pyright` (no flags) reports
   `reportMissingImports` for numpy / pandas / scipy / statsmodels / pytest
   because `pyrightconfig.json` (owned by P0) doesn't set `venvPath` /
   `venv`. With `--pythonpath .venv/bin/python` it's clean (0 errors).
   Recommend P0 / integration adds:
   ```json
   "venvPath": ".",
   "venv": ".venv"
   ```
   I did not touch the config because it's outside my workspace.

2. **`materialise_design` lives inside the assumptions package**, not in
   P1.A's `_internal/design.py`. Once P1.A's design-builder is merged,
   `_common.materialise_design` should delegate to it so basis-expansion
   terms work for prerequisite/VIF/classical checks at every phase. This
   was the only sensible way to avoid blocking on P1.A.

3. **Stability checks are CV-stubbed for now** (per the AGENTS.md
   contract). The CV-backed paths are written and lightly inferred from
   `cv.coefficients` / `cv.metrics` / `cv.fold_results` shapes, but P3.B
   will pin the real `CVResult` interface — expect a small adapter pass
   when P3 integrates.

4. **`LowVIF.suggestion` exact string** is locked behind a test. If anyone
   touches the wording, that test fails — by design.

5. **`Assumption` protocol uses `@property` declarations** for the four
   attributes (`name`, `severity`, `requires_solution`, `requires_cv`)
   so that frozen dataclasses with class-attribute defaults satisfy the
   protocol under pyright's strict variance rules. Implementers can still
   declare them as plain dataclass fields with defaults — pyright accepts
   the implicit getters as property-compatible. No behavioural impact.
