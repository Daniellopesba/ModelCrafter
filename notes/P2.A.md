# Task P2.A — Penalty primitives (l1, l2, PenaltySum) + ComparableFeatureScales

## What got built

The Phase 2 penalty algebra. `model_crafter.penalty` now exposes `l1(lam)`,
`l2(lam)`, the concrete `L1Penalty`, `L2Penalty`, and `PenaltySum` frozen,
slotted dataclasses, alongside the P1.A `NoPenalty` / `Penalty` protocol
(both kept; nothing renamed). `L1Penalty.value` is `lam * sum(|beta_j|)`;
its `prox(beta, step)` is element-wise soft-thresholding
`sign(beta_j) * max(|beta_j| - lam*step, 0)`. `L2Penalty` uses the glmnet
half-norm `(lam/2) * sum(beta_j^2)`; its prox is shrinkage
`beta_j / (1 + lam*step)`. Constructors validate `lam >= 0` and finite.
`+` between any two penalties returns a flat `PenaltySum` (composition of
`(l1+l2)+l1` produces a three-part sum, not a nested one). `+` with
`NoPenalty` returns the non-identity operand; `+` with a `Term` raises
`TypeError` with the documented `features=`/`penalty=` redirect message.
`PenaltySum.value` sums part values; `PenaltySum.prox` detects the
elastic-net (L1 + L2) structure and applies the Parikh-Boyd §6.5.2 closed
form `soft_threshold(beta / (1 + l2*step), l1*step)`; multiple L1 / L2
parts are combined (lambdas summed) before composition, and any unknown
additional parts are then composed by their own proxes. `PenaltySum.assumptions`
deduplicates equal assumption instances so the framework doesn't run the
same `ComparableFeatureScales` check twice.

`model_crafter.assumptions.stability` gained `ComparableFeatureScales` (SOFT,
`requires_solution=False`, `requires_cv=False`, default `std_ratio_max=100`).
It materialises the design via the existing `_common.materialise_design`,
takes the per-column std with `ddof=0`, excludes the `(Intercept)` column
and any column with zero std, then compares `max(std)/min(std)` to the
threshold. Message: `"feature std ratio = {ratio:.1f} (threshold
{std_ratio_max:.1f}); consider scaling features before applying L1/L2."`
Suggestion (on fail): `"Standardise features (zero mean, unit std) before
fitting penalised models — L1/L2 are scale-sensitive (ESL §3.4.1)."`
`model_crafter/assumptions/__init__.py` re-exports the new symbol and
adds it to `__all__`.

criterion the task lists. Highlights:

- Acceptance #1 (`l1(0.5) + l2(0.5)` yields a 2-part `PenaltySum`, iter
  preserves order, `len==2`): `test_l1_plus_l2_is_penalty_sum_with_two_parts`.
- Acceptance #2 (`ComparableFeatureScales` fires when feature std ratio
  > 100): `test_comparable_feature_scales_fires_when_scales_wildly_different`
  exercises the assumption directly; the framework-level variant
  `test_comparable_feature_scales_fires_via_run_assumptions` confirms
  `run_assumptions` collects the assumption from `spec.penalty.assumptions`,
  records the failing SOFT result, and emits a warning under the default
  `on_violation` path. (See "Open notes" below for why the end-to-end
  `mc.solve` variant isn't in this task.)
- Acceptance #3 (`value()` and `prox()` match hand-derived references to
  1e-12 for L1, L2, and elastic net across random `(beta, step, lam)`
  triples): `test_l1_value_property_random`, `test_l2_value_property_random`,
  `test_l1_prox_property_random`, `test_l2_prox_property_random`,
  `test_elastic_net_prox_property_random` (50 random triples each).

Sum+Penalty); equivalence of `l1+l2` vs `l2+l1` for elastic-net prox;

## Stubs / mocks / open questions

* The end-to-end `mc.solve` acceptance variant ("unscaled features
  trigger the `ComparableFeatureScales` warning by default" from §8 Phase
  2) is **not** wired in this task because the ridge/lasso solvers are
  P2.B and P2.C respectively. `solve()` currently raises
  `LookupError: no solver registered for (loss=_SquaredErrorLoss,
  penalty=L2Penalty)` when handed an L2 penalty. P2.INTEG (after P2.B
  lands) is the natural place to add a `tests/integration/test_phase2.py`
  case that runs `mc.solve(spec=…, penalty=l2(λ))` on wide-scale data and
  catches the warning. The framework-level test
  `test_comparable_feature_scales_fires_via_run_assumptions` confirms
  the warning machinery itself works.

* The `Penalty` protocol grew a `prox(beta, step) -> ndarray` method
  (DESIGN.md §6.1 always listed it; P1.A omitted it because `NoPenalty`
  didn't need it). I added an identity-map `NoPenalty.prox` so the
  protocol contract is satisfied. This is a *widening* of the P1.A
  interface, not a breaking change: existing call sites that only used
  `value` still work. Flagging here in case the integration agent
  prefers to keep `prox` off the protocol and let solvers
  `isinstance(p, L1Penalty)`-dispatch instead.

* `ComparableFeatureScales` lives in `stability.py` per DESIGN.md §7's
  file layout (SOFT lives in `stability.py`). Materialising the design
  inside the check is unavoidable in Phase 2 because the assumption runs
  *before* any solver builds its own design matrix. Once P4.B introduces
  WoE/binned terms, `materialise_design` in `assumptions/_common.py`
  will need a richer fit-state plumbing path; the check itself is
  agnostic to that. No design change requested.

* `PenaltySum.parts` is declared as `tuple` (not `tuple[Penalty, ...]`)
  to keep the frozen-dataclass + slots story working cleanly under
  pyright basic mode. Each part is validated in `__post_init__` for
  `value` + `__add__`, and nested `PenaltySum`s are rejected (the
  flattening invariant). Future strict-mode typing can tighten this.

* `PenaltySum.prox` uses the elastic-net closed form when parts are all
  L1/L2/NoPenalty; for unknown atomic penalty types it falls back to
  operator splitting (combined-L1/L2 first, then the unknown part's
  prox). That's the conservative split P2.C's coordinate descent needs.

* Workspace expansion: I only touched the four files the task listed
  (`model_crafter/penalty.py`, `model_crafter/assumptions/stability.py`,
  `model_crafter/assumptions/__init__.py`, `tests/test_penalty.py`).
  Nothing else changed.
