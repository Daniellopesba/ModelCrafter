# Task P4.A — Standard basis expansions

## What's built

New module `model_crafter/terms/basis.py` providing the six basis-term
factories pinned in `AGENTS.md`:

| Function   | Returns                | Math                                          |
| ---------- | ---------------------- | --------------------------------------------- |
| `ns(col, df, *, knots=None)`            | `_NSTerm`     | Natural cubic spline (ESL §5.2.1) — B-spline basis projected onto the natural BC null space, with the constant direction removed. Linear extrapolation beyond the boundary knots is implemented explicitly. |
| `bs(col, df, *, degree=3, knots=None)`  | `_BSTerm`     | B-spline (Cox / De Boor recursion via `scipy.interpolate.BSpline`). Leading column dropped to match R's `bs(..., intercept=FALSE)`. |
| `poly(col, degree)`                     | `_PolyTerm`   | Orthogonal polynomial basis — QR of the centred Vandermonde, constant column dropped (matches R's `poly()`). |
| `step(col, breakpoints)`                | `_StepTerm`   | Piecewise-constant indicators, reference bin dropped. |
| `smooth(col, df)`                       | alias for `ns` | Per DESIGN.md §2.5, true roughness-penalty smoothing splines are out of scope for v0; `smooth` is exactly `ns`. |
| `hinge(col, knot, direction)`           | `_HingeTerm`  | MARS-style $(t-x)_+$ or $(x-t)_+$ (ESL §9.4.1). |

Each term is a `@dataclass(frozen=True, slots=True)` implementing the
`Term` protocol (`name`, `expand`, `__add__`, `__radd__`). Each term
declares `assumptions = (SupportContainsPredictData(),)` so the
assumption framework runs the extrapolation guard when predicting.

### State round-trip

I added an internal `_BasisExpandedTerm(ExpandedTerm)` subclass that
carries a `state: Mapping[str, Any]` field alongside the protocol's
`columns` / `values`. This is type-compatible with everything
`_internal/design.py` reads (which only touches `.columns` and
`.values`) and gives basis terms a way to return their learned knots /
QR / projection without modifying `terms/base.py`.

At fit time: `expand(data, fit_state=None)` computes the knots /
orthogonalisation from the *training* `data[col]` and returns the
expansion together with the learned state.
At predict time: `expand(data, fit_state=<learned>)` uses the stored
state to evaluate the basis on `data`, so columns line up with training.

### `SupportContainsPredictData`

Added additively to `model_crafter/assumptions/stability.py` (and
re-exported from `assumptions/__init__.py`). SOFT severity,
`requires_solution=True`, `requires_cv=False`. At predict time it walks
`sol.fit_state` for each basis term, reads
`fit_state[term.name]["boundary_knots"]`, and computes the fraction of
predict-time rows outside that range. The message names the column, the
fraction, and the training range
(`"feature 'x': 30.0% of rows outside training knot range [0, 10]"`).

## Tests

`tests/test_basis.py` — 31 tests, all green. Verification routes:

* **`bs`** matches `patsy.splines.bs` (R-equivalent) to 1e-10 on cubic
  and degree-4 splines.
* **`ns`** matches an independent B-spline + null-space projection
  reference (also bundled into the test file) to 1e-10, on random and
  uniform `x`. Linear extrapolation beyond boundary knots is verified
  to 1e-10 by fitting a line to each basis column on the extrapolation
  region. A linear polynomial is recovered exactly (it's in the natural
  spline space) and the basis is self-consistent under fit→predict
  round-trip.
* **`poly`** matches a hand-derived QR-based reference (R's
  `stats/R/poly.R` algorithm) to 1e-12. Orthonormality of columns is
  verified directly; a degree-3 polynomial is recovered to 1e-10.
* **`step`** with three breakpoints produces four bins whose
  re-aggregated indicator columns sum to 1 per row.
* **`hinge`** matches the closed-form expressions for both directions.
* **`SupportContainsPredictData`** fires (passes / fails) at the right
  thresholds; the message names the fraction; with `solution=None` it
  returns a skipped-pass result. All Phase-4 basis terms declare the
  assumption in their `assumptions` tuple.

## `_internal/design.py` — integration gap flagged

`build_design` currently constructs an empty `next_fit_state: dict[str, Any] = {}`
and writes it to `DesignMatrix.fit_state` without ever populating it.
The solvers (OLS / ridge / coordinate / IRLS) each write
`fit_state=MappingProxyType({})` into their `SolverOutputs`, and
`solve()` propagates that to `Solution.fit_state`. As a result,
`Solution.fit_state` is empty for basis-term specs even though the
terms learn knots / QR / projection at fit time.

The basis terms in this task **can** round-trip state if `fit_state`
arrives, but in the current flow it never does. The right fix is small
and surgical:

1. In `_internal/design.py`, after each `term.expand(data, fit_state=...)`,
   if the returned `ExpandedTerm` has a `state` attribute (i.e. it's
   `_BasisExpandedTerm`), capture it into `next_fit_state[term.name]`.
   `DesignMatrix.fit_state` is already on the return type — just
   populate it. This is two lines.

2. In each solver (`ols.py`, `ridge.py`, `coordinate.py`, `irls.py`),
   change `fit_state=MappingProxyType({})` to
   `fit_state=inputs.design.fit_state`. The `SolverInputs.design` field
   already carries the `DesignMatrix`, so the solver only needs to
   forward it.

3. Optionally: tighten `ExpandedTerm` itself (in `terms/base.py`) to
   include an optional `state` field. That would let
   `_internal/design.py` use `getattr(exp, "state", None)` cleanly.
   I did **not** do this because `terms/base.py` is outside this
   task's workspace; `_BasisExpandedTerm` (a subclass) is the
   light-touch alternative.

End-to-end `mc.solve / mc.predict` on a spec with `mc.ns(col, df=k)`
will not round-trip the learned state until the integration agent
makes the small changes above. P4.A's tests cover the round-trip via
direct `term.expand(..., fit_state=...)` calls — i.e. the basis-side
contract is verified; only the propagation through `solve` is
unwired.

## Open questions

* **Naming convention for basis columns.** I chose
  `bs(x)_0, bs(x)_1, ...`, `ns(x)_0, ...`, `poly(x)_1, poly(x)_2, ...`
  (poly is 1-indexed because column 0 is the dropped constant), and
  `step(x)_bin1, step(x)_bin2, ...`. The integration agent should
  decide whether to lock this in or rename to match patsy/R
  conventions exactly. None of the package's existing code depends on
  the spelling.

* **`smooth` semantics.** I implemented it as a pure alias for `ns`.
  When proper smoothing splines (penalised, with a roughness
  parameter) ship in a later phase, the `smooth` factory should
  probably stop pointing at `ns` and start returning a distinct term
  type with a roughness-penalty assumption.

* **Public re-exports.** `model_crafter/terms/__init__.py` is off-limits
  for this task; the integration agent will wire `mc.ns`, `mc.bs`,
  `mc.poly`, `mc.step`, `mc.smooth`, `mc.hinge` into the package's
  public API.
