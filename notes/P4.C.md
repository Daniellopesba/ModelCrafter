## What I built

- `model_crafter/terms/interact.py`: the `interact(*cols)` and `cross(*cols)`
  factories plus the single frozen dataclass `InteractionTerm` that backs
  both. `kind` âˆˆ `{"interact", "cross"}` discriminates between
  main-effects-plus-interactions and interaction-only expansions. The
  factories
  - auto-promote string operands to `RawTerm` via the public
    `terms.base._promote`,
  - flatten same-kind nested operands on construction
    (`interact(interact(a, b), c) == interact(a, b, c)`,
    `cross(cross(a, b), c) == cross(a, b, c)`),
  - keep mixed-kind nesting opaque (`cross(interact(a, b), c)` treats the
    inner `interact` as one multi-column basis),
  - reject `len(operands) < 2` with a clear `ValueError`.
- The expansion uses a Khatriâ€“Rao (row-wise Kronecker) helper. For
  `interact` the result block-concatenates: main effects in operand order,
  then all 2-way Khatriâ€“Rao products (lex `(i, j)` with `i < j`), then
  3-way, ..., up to the full all-operands product. For `cross` the result
  is only the highest-order Khatriâ€“Rao product. Column names are
  deterministic: `*`-joined sub-column names (`x*y`, `ns(x)_1*z`,
  `woe(x)*income`, `x*y*z`, etc.).
- Assumptions: `InteractionTerm.assumptions` is the union of operand
  assumptions, preserving operand order, deduplicated by object identity
  so the framework doesn't run a shared check twice. Interaction terms
  declare no new assumptions of their own. Documented in the module
  docstring with ESL Â§5 (basis interactions) and Â§9.4 (MARS products) as
  references.
- `fit_state` pass-through: `expand(data, fit_state=...)` reads a
  `{"operands": {operand_name: state, ...}}` mapping and threads each
  slice into the corresponding operand's own `.expand` call. `None`
  (training-time) and missing keys both result in `fit_state=None` for
  the operand. This is forward-compatible with whatever shape P4.A's
  `ns`/`bs`/`poly` and P4.B's `woe`/`binned` decide to expose; the
  interaction wrapper makes no assumption about what an operand stuffs
  into its state.

## Open questions / contract concerns

- **`fit_state` shape is my call.** AGENTS.md pins the public functions
  (`interact`, `cross`) but does not pin the per-term `fit_state` shape.
  I chose `{"operands": {operand_name: state, ...}}`. This survives
  P4.A's basis terms and P4.B's WoE terms as long as their own
  `fit_state` slots are JSON-shaped dicts keyed however they like â€” the
  interaction wrapper is opaque about contents. **Contract concern for
  the integration agent**: when wiring `_internal/design.py` to capture
  fit_state from `ExpandedTerm` (currently the field on `ExpandedTerm`
  doesn't exist; `RawTerm` is stateless), the per-term mapping for an
  interaction must merge sub-states under the `"operands"` key. If P4.A
  or P4.B picks a colliding key, this is the place to reconcile.
- **No assumptions field on `ExpandedTerm`.** Per the Term protocol,
  `assumptions` lives on the term object (it's a class-level / property
  on the Term, not on the expansion). I exposed it as a `@property` on
  `InteractionTerm`. If the assumption-collection driver (P1.B's
  framework â€” see `assumptions/__init__.py`) walks `spec.features` and
  iterates `term.assumptions`, this just works. If it walks
  `ExpandedTerm.assumptions` (no such attribute exists today), the
  framework needs the property-on-term hook. Worth confirming at
  P4.INTEG.
- **Column-name collisions.** `interact("x", "x")` would produce columns
  `x, x, x*x` and `build_design`'s duplicate-name guard
  (`_internal/design.py` line ~125) would correctly raise. I did not
  add a separate guard in `InteractionTerm` because the existing one is
  sufficient and centralised. If callers expect a friendlier message
  ("you passed `x` twice to interact"), that's a small follow-up.
- **Self-products.** `cross("x", "x") = x*x` is allowed in some
  packages as a way to express `poly("x", 2)`. I treat it as a normal
  Khatriâ€“Rao step (yields one column named `x*x`), so it works but the
  preferred spelling is `mc.poly("x", 2)` (P4.A). Documented in the
  module docstring implicitly via "products of basis functions".
- **Stubs for P4.A/P4.B.** Tests use local `_NsStub` (multi-column
  basis-like) and `_WoEStub` (single-column stateful) implementing the
  Term protocol. They are pure-stdlib, ~30 lines total, and the
  integration agent should replace them with `from
  model_crafter.terms.basis import ns` and `from
  model_crafter.terms.woe import woe` at P4.INTEG once those land. The
  stubs are intentionally named with a leading underscore and live in
  the test file (not in the package) so the replacement is mechanical.
- **`mc.binned` mention in Acceptance #3.** The AGENTS.md acceptance
  criterion mentions "categorical (one-hot)" terms. P4.B's `binned`
  isn't merged. I tested a 0/1 numeric indicator column (`cat` in the
  toy frame) as a stand-in: it exercises the
  continuous-Ã—-binary-indicator math which is the meat of the
  categorical case. Once `binned` is available the integration test
  can add a `binned Ã— continuous` case for completeness; the math is
  identical at the Khatriâ€“Rao layer.
