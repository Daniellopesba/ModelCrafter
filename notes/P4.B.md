## Math captured in docstrings

Per the task brief and DESIGN.md §3.1: WoE with Laplace smoothing
(`+0.5` per bin per side), Information Value
`IV = Σ_b (p_event_b − p_nonevent_b) · WoE_b`, the
"useless / weak / medium / strong / leakage" IV rules of thumb. The four
binning algorithms (monotonic greedy merge, single-feature CART, rare-
category grouping, manual edges) are derived in the module docstring and
each strategy's class docstring.

## How I handled missing values and unseen levels

Per DESIGN.md §11 (open questions resolved):

- **Missing values at fit time:** every WoE/binned term creates an
  explicit `(Missing)` bin if any NaN is seen in the source column at
  fit time. The `(Missing)` bin is the *last* entry in
  `result.bin_labels / n_events / n_nonevents / woe` so numeric bin
  ordering is preserved for the monotonicity check (which excludes the
  `(Missing)` bin per DESIGN.md §3.1).
- **NaN at predict time, no `(Missing)` bin learned:** route to bin 0
  with a warning (so the failure is loud).
- **Unseen categorical levels:** assign WoE = 0 (neutral evidence) and
  warn — exactly the DESIGN.md §11 lean.
- **Numeric values outside training range at predict time:** fall into
  the nearest edge bin (closed-on-both-sides extrapolation). This is the
  natural consequence of the bin definition being open-on-left,
  closed-on-right with `±∞` padding.

## Integration with `_internal/design.py` — the flagged extension

`_internal/design.py` *does* support multi-column terms (`ExpandedTerm`
already carries a 2-D `values` array and `build_design` `np.hstack`'s
expansions). But it has two gaps relevant to WoE:

1. **Stateful (supervised) terms.** `term.expand(data, fit_state=...)`
   does not receive `y`. WoE/binned binning is supervised — bins are a
   function of `(x, y)`. The current solver dispatch (`solve/__init__.py`)
   does not have a hook to fit terms with `y` before calling
   `build_design`, and `build_design` does not pass `y` to terms.

2. **State capture is one-directional.** `ExpandedTerm` carries `columns`
   and `values` but no `state`. The local variable `next_fit_state` in
   `build_design` is initialised to an empty dict and never populated.
   That means even if a term *could* learn state during `expand`, the
   state would not round-trip to `Solution.fit_state`.

**My workaround**: terms are values, and a *fitted* binning term is a
value that carries its bins on the term itself (the `BinningResult` lives
on `WoETerm.fitted` / `BinnedTerm.fitted`). The user calls
`fit_binnings(spec, data)` to produce a new `LinearSpec` whose binning
terms are pre-fit; `expand()` reads bins off the term, not off
`fit_state`. The flow today is:

```python
spec = mc.linear(target="y",
                 features=mc.woe("x", bins=mc.monotonic()),
                 loss=mc.logistic,
                 penalty=mc.l2(0.1))
spec = mc.fit_binnings(spec, df)              # NEW: supervised pass
sol = mc.solve(spec, df)                       # plain Phase-3 solve
```

**For the integration agent** (P4.INTEG): once `_internal/design.py`
grows a state-capture mechanism (the cleanest extension is adding a
`state: Mapping = MappingProxyType({})` field to `ExpandedTerm` plus
populating `next_fit_state[term.name] = exp.state` in
`build_design`), `solve()` can be wrapped to call `fit_binnings(spec,
data)` automatically and thread the resulting state through the existing
`Solution.fit_state` plumbing. The public `fit_binnings` helper is
designed to be the bridge; it is also the right user-facing API even
after auto-wiring (lets users inspect or override the binning before
solving).

**A second integration gap unrelated to WoE state**: the assumption
framework's own `materialise_design` in `assumptions/_common.py` treats
each feature as a single column referenced verbatim from `data`. For
WoE-encoded terms this is wrong: the assumption-time design matrix uses
the *raw* `x` column where the solver uses the *WoE-encoded* values.
The post-fit `NoPerfectSeparation` check therefore mis-fires for any
WoE-logistic spec (it computes `η = X_raw @ β_WoE`, sees enormous
magnitudes, and flags saturation). For now the tests pass via
`suppress=(NoPerfectSeparation,)`; the integration agent should swap
`materialise_design` for the real `_internal/design.build_design` once
that supports stateful terms.

## Anything stubbed

Nothing functionally — every method on every public type has a real
implementation. Two minor points worth flagging:

- `WoETerm` / `BinnedTerm` declare `assumptions: tuple[Any, ...]` with
  a `field(default_factory=tuple)` and override via
  `object.__setattr__` in `__post_init__`. The pattern is forced by
  Python's dataclass restriction on mutable defaults containing
  references to forward-declared assumption classes, but the runtime
  shape is identical to the other Phase-3 terms.

  rich `_repr_html_` is a Phase-6 polish task per DESIGN.md §8.
