# Task P3.B — Temporal validation: splitters, CV, tune, nested_cv

## What was built

* **Splitters** (`model_crafter/validation/splitters.py`)
  * `time_split(df, time_col, ratios)` — single chronological split into
    `len(ratios)` contiguous slices.
  * `expanding_window(time_col, n_folds, horizon, gap, min_train=None)` —
    walk-forward, growing training window.
  * `rolling_window(time_col, train_size, horizon, step, gap)` —
    fixed-width sliding window.
  * `purged_kfold(time_col, n_folds, gap)` — López de Prado-style
    purged k-fold (equal-row buckets in time order; train excludes
    the purge zone `[v_lo - gap, v_hi + gap]`).
  * Each concrete splitter is a frozen dataclass exposing `time_col`,
    `gap`, and a `.split(df)` iterator. The `Splitter` protocol is
    runtime-checkable.
  * Validation is eager: invalid `n_folds`, negative `horizon`/`gap`,
    or unparseable Timedelta strings raise at construction (DESIGN.md
    §9.8).

* **`NoTemporalLeakage`** (`model_crafter/assumptions/temporal.py`)
  * HARD assumption. Reads `cv.splitter.time_col` and
    `cv.splitter.gap`, walks `cv.folds`, and verifies
    `max(train[time_col]) + gap <= min(valid[time_col])` for every
    fold.
  * Skips with `passed=True` when `cv is None` (per the framework's
    standard "no CV available" pattern).
  * Wired additively into `assumptions/__init__.py` `__all__`.

* **`cross_validate`** (`model_crafter/validation/cross_validate.py`)
  * Assessment-only runner. Iterates `splitter.split(df)`, fits the
    spec via `solve(...)` on each train fold, scores `metrics` on the
    validation fold, and returns a frozen `CVResult` with
    `fold_results` (one dict per fold), `solutions`, the original
    `splitter`, and an `AssumptionReport` carrying a
    `NoTemporalLeakage` check.
  * Refuses to run when `spec.loss.label_horizon` is set and the
    splitter's gap is zero or smaller than the horizon (DESIGN.md
    §3.2). Implemented defensively with `getattr(...)` so it works
    today against squared-error (no `label_horizon`) and against
    P3.A's `LogisticLoss` (which will set one) without changes.
  * Defensively re-checks the gap condition per fold *before*
    solving so a custom splitter that ignored its own gap also fails
    fast.

* **`tune` + selection rules** (`model_crafter/validation/tune.py`)
  * `tune(spec_fn, grid, data, splitter, metric, weights=None,
    rule=None, direction="maximize")` evaluates each grid value via
    `cross_validate`, builds a `cv_curve` DataFrame (indexed by grid
    value with columns `metric_mean` / `metric_sd`), applies the
    selection rule, and refits on the full data at `best_param`.
  * `best_mean(curve, direction)` — argmax / argmin of `metric_mean`.
  * `one_se_rule(curve, direction)` — ESL §7.10 parsimony rule;
    picks the largest index value (i.e., largest lambda) whose
    `metric_mean` is within 1 SE of the best.
  * `TuneResult.solution.spec.penalty.lam == best_param` is exercised
    in the test suite.

* **`nested_cv`** (`model_crafter/validation/tune.py`)
  * `nested_cv(spec_fn, grid, data, outer_splitter, inner_splitter,
    metric, ...)` — on each outer fold, runs `tune` on the outer
    train with the inner splitter, refits at the inner-chosen param
    on the outer train, scores on the outer test. Returns
    `outer_metric` (DataFrame), `best_params` (tuple — stability
    diagnostic), and `inner_curves` (tuple of curves).
  * The acceptance test runs **50 small-sample simulation replicates**
    (n=400, 8 features, high noise, 3+3 folds) and confirms the
    optimism bias is detectable with a one-sided paired t-test
    (p < 0.05) on `(tune.cv_curve.loc[best, mean] - nested.outer_mean)`.

* **`over_time`** (`model_crafter/validation/over_time.py`)
  * Time-indexed metric runner. Returns a `pd.Series` keyed by the
    midpoint of each validation window.

* **Validation `__init__.py`** extended additively to re-export
  `Splitter`, `CVResult`, `TuneResult`, `NestedCVResult`,
  `time_split`, `expanding_window`, `rolling_window`,
  `purged_kfold`, `cross_validate`, `tune`, `nested_cv`,
  `best_mean`, `one_se_rule`, `over_time`. `lambda_path` / `log_grid`
  preserved exactly.

## What's tested

* `tests/test_splitters.py` (16 tests) — chronological order, gap
  enforcement, train-size invariants, argument validation. The
  expanding-window acceptance criterion (gap respected on a synthetic
  daily panel) is in `test_expanding_window_respects_gap_acceptance`.
* `tests/test_cross_validate.py` (10 tests) — happy path, `summary()`
  shape, weights support, label_horizon refusal (gap=0 ⇒ raise),
  label_horizon-ok-with-nonzero-gap, NoTemporalLeakage direct check
  (pass on clean, fail on manual violation, fail on gap violation,
  skip-when-cv-None, attached to `CVResult`).
* `tests/test_tune.py` (10 tests) — both selection rules on synthetic
  curves with known optima; end-to-end `tune` returns `TuneResult`,
  best_mean acceptance criterion, one_se_rule acceptance criterion;
  argument validation.
* `tests/test_nested_cv.py` (4 tests) — shape, optimism-bias
  acceptance (50-replicate simulation, paired t-test, p<0.05),
  one_se_rule path, argument validation.

258 tests total pass (218 baseline + 40 new). `ruff check`, `pyright`,
and `pytest` all green.

## Stubs / assumptions about parallel work

* **Metric primitives (P3.D).** The test suite uses inline metric
  callables (`_mse_metric`, `_neg_mse`) returning bare floats. The
  runner accepts both bare floats and any object with a `.value`
  attribute, so P3.D's richer `AUCResult`/`KSResult`/etc. plug in
  without changes to `cross_validate` / `tune` / `nested_cv`.
* **Logistic loss `label_horizon` (P3.A).** P3.A is expected to set
  `LogisticLoss.label_horizon = pd.Timedelta(...)` for credit
  modelling. `cross_validate` reads this via `getattr(spec.loss,
  "label_horizon", None)`. The tests monkey-patch `label_horizon`
  onto the `squared_error` singleton (with cleanup) to exercise the
  refusal path defensively.
* **Stability assumptions (P3.INTEG).** `CoefficientStability` and
  `PredictiveStability` (declared by `squared_error.assumptions`)
  read fold solutions through duck-typed shapes already implemented
  in P1.B's `_real_coefficient_stability` /
  `_real_predictive_stability`. They are not yet wired into
  `cross_validate`'s assumption report — P3.INTEG can add them by
  extending the `audit_cv` shape to expose `.fold_results` (already
  present) and calling `run_assumptions` with that as the `cv=`.
  Left for integration so the responsibility is in one place.

## Contract decisions worth flagging

* `expanding_window.min_train` defaults to one `horizon` when not
  passed — the first fold trains on at least that much history.
  AGENTS.md leaves the default unspecified.
* `purged_kfold` partitions by **row count** (equal-size buckets in
  time order), not by time. This keeps validation fold sizes
  comparable when timestamps are not uniformly distributed; an
  alternative time-based partition is straightforward to add if a
  caller needs equal-time buckets.
* `one_se_rule`'s "simpler = larger grid value" convention is
  hard-coded — appropriate for L1/L2 lambdas. Callers wanting a
  different parsimony order can pre-sort the curve before passing
  it; the rule walks `eligible.index.max()` which respects pandas
  index ordering.
* `cross_validate` raises `AssumptionError` on the label_horizon
  refusal rather than warning — this is a HARD-severity violation
  in spirit (a leaky CV is invalid) so a hard refusal matches
  DESIGN.md §3.2's intent.
