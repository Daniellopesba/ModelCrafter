# P5.B — Segmented performance

## What was built

`model_crafter.performance.by_segment` exposes:

* `SegmentedPerformanceReport` — a frozen, slotted dataclass with
  `segments: dict[str, PerformanceReport]` and
  `aggregate: PerformanceReport`. Its `__repr__` opens with the aggregate
  header (n / events / AUC / KS / Brier) and then a one-line summary per
  segment, sorted on the stringified key for stable ordering.
* `performance_by_segment(sol, data, by, *, weights=None, reference=None)`
  — pure orchestration over `model_crafter.performance.performance`. For
  each `data.groupby(by, observed=True)` slice it calls `performance(sol,
  slice_, weights=weights, reference=reference)`; the aggregate is the
  unsegmented `performance(sol, data, ...)` call.

This is the deployment-monitoring counterpart to `mc.performance_over_time`
(P5.A): one fitted `Solution`, split the held-out evaluation frame.
Re-fitting per segment is `mc.segmented` in Phase 6 — explicitly out of
scope here, documented in the module docstring.

## Design choices, called out

1. **Empty segments are dropped silently.** `groupby(..., observed=True)`
   handles Categorical levels with zero rows, which is the only way an
   empty segment can sneak in. Computing a `PerformanceReport` over an
   empty frame would be meaningless (zero-division on event rates, NaN
   AUC) and reporting it adds noise to the repr.
2. **Numeric keys → strings.** `groupby` hands back numpy scalars; we
   coerce them via `str(key)` so the dict has stable, sortable, JSON-
   friendly keys downstream (e.g. for `pd.DataFrame.from_dict`).
3. **`reference` passes straight through** to every per-segment
   `performance` call. That means the PSI sub-report on a segment
   measures drift of that segment's predicted scores against the *whole*
   reference distribution — the same reference all segments see. If we
   later want per-segment reference frames, that's a new kwarg
   (`reference_by_segment=`), not a behaviour change.
4. **Unknown `by` raises `KeyError`** with the bad column name in the
   message — failing fast at the orchestration boundary, before we hit
   pandas' less specific error.

## What's tested

`tests/test_performance_by_segment.py` — 11 tests covering:

* **Acceptance criterion (Phase 5 §8 #4 + AGENTS.md P5.B):**
  - `test_segment_n_obs_sum_to_aggregate` — `Σ segments[k].n_obs ==
    aggregate.n_obs == len(test)`.
  - `test_segment_auc_matches_manual_slice` — per-segment AUC equals
    `mc.auc(sol, test[test[by] == k]).value` to **bitwise** equality
    (`abs_tol=0, rel_tol=0`). The orchestrator is doing exactly what a
    user would do by hand.
* **Aggregate consistency** — `aggregate` matches `mc.performance(sol,
  data)` on `n_obs`, `n_events`, AUC, and Brier bitwise.
* **Structural** — return type, frozen + slots, dict-of-PerformanceReport.
* **Numeric segment keys** — integer-typed `by` column yields string
  dict keys `{"0", "1", "2"}`.
* **Weights pass-through** — uniform weights match unweighted bitwise
  per segment.
* **Reference pass-through** — supplying `reference=` lights up the PSI
  `stability` sub-report on every segment and on the aggregate.
* **Empty segments dropped** — a Categorical level with zero rows does
  not appear in `segments`.
* **Repr structure** — opens with `SegmentedPerformanceReport`, mentions
  the aggregate, names every segment key, and shows the per-line AUC /
  KS / Brier metrics.
* **Unknown `by`** — `KeyError` naming the bad column.

Logistic fitting comes from `mc.logistic` (P3.A, already merged on
phase-5 baseline), so this test exercises the real end-to-end path with
no mocks.
